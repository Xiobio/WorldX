# WorldX Simulation Behavior

## Core model

Keep these boundaries intact unless the task explicitly changes architecture:

- Server owns logical state:
  - day and tick
  - logical location / mainAreaPointId
  - action decisions (with optional innerMonologue)
  - dialogue, memories, relationships, effects
  - timeline recording (events.jsonl, state.db)

- Client owns local presentation:
  - pathfinding and path smoothing
  - dialogue approach animation (initiator walks to responder)
  - ambient idle wandering near the current anchor
  - playback timing, speech/thought bubbles, scene presentation
  - replay mode (client-driven playback from JSONL)
  - day transition overlay (multi-phase: ending → starting → fade-out)

Do not add persistent server-side `x/y` state unless the task clearly needs an architectural change.

## Run mode vs Replay mode

The UI provides a segmented mode toggle: **Run** / **Replay**.

- **Run mode** ("live"): server-driven simulation. Play/Pause sends ticks to the server. All features (God Panel, Sandbox Chat, New Timeline, world/timeline switching) are available.
- **Replay mode** ("replay"): client-driven playback from `events.jsonl`. Play/Pause advances through recorded frames. Server-mutating features are disabled.

`PlaybackController` tracks `mode: "live" | "replay"`:
- `set_auto_play` event is routed to `setAutoPlay()` (live) or `setReplayAutoPlay()` (replay) by `WorldScene.ts`
- replay reads frames from the timeline API and emits events locally
- replay emits `replay_progress` events for the progress bar
- when replay finishes, `replay_finished` fires and the UI auto-switches back to Run mode

### Replay guards

During replay, these server-mutating calls must be skipped:
- `apiClient.patchCharacterRuntimeState` in `WorldScene.handleDialogue`
- any direct simulation API calls

`WorldScene` tracks `isReplaying` state and checks it before server calls. Events like `replay_init` position characters at their initial state when replay begins.

Key files:
- `client/src/systems/PlaybackController.ts`
- `client/src/scenes/WorldScene.ts`
- `client/src/ui/panels/TopBar.tsx`
- `client/src/ui/App.tsx`

## Inner monologue

Characters can have inner thoughts (`innerMonologue`) on both action decisions and dialogue turns. These are:
- Generated by the LLM as optional fields in the action decision schema and dialogue turn schema
- Carried on `SimulationEvent.innerMonologue`
- Displayed as cloud-shaped thought bubbles on the client, styled differently from dialogue bubbles
- Shown in character history logs
- During dialogue, thought bubbles stack above dialogue bubbles rather than replacing them

The frequency of inner monologue is controlled by prompt wording — it should be infrequent but dramatic/interesting when it occurs.

Key files:
- `server/src/simulation/decision-maker.ts` — action decisions with optional innerMonologue
- `server/src/simulation/simulation-engine.ts` — propagates innerMonologue on dialogue turn events
- `server/src/llm/output-schemas.ts` — schema definition
- `client/src/objects/CharacterSprite.ts` — thought bubble rendering

## Location and movement model

Current rules:
- regions come from the `regions` layer
- interactive elements come from `interactive_objects`
- region-less worlds still work through synthetic `main_area`
- large `main_area` spaces use generated `mainAreaPoints`
- each interactive element gets an `element_<id>` approach point
- server decides meaningful movement; client converts that into walkable pixel targets
- characters avoid standing on walkable area edges (configurable clearance)

Movement intent:
- `move_to` and `move_within_main_area` are real server decisions
- client ambient wandering is only micro-movement after arrival
- anchored characters do not receive long-distance move actions
- point-to-point movement in `main_area` stays logical on the server and presentation-only on the client
- dialogue spacing scales from character display size, not absolute pixels
- walking speed scales from relative character/map size, not fixed raw pixels

Path smoothing:
- EasyStar paths go through staircase rebalancing plus L-shaped shortcutting
- each optimization still verifies walkability before committing

- `server/configs/prompts/reactive-decision.md`
- `client/src/systems/CharacterMovement.ts`

## Character anchoring

Characters may have `anchor: { type: "region"|"element", targetId }`.

Server-side enforcement:
- `action-menu-builder.ts` omits `move_to` and `move_within_main_area`
- `action-executor.ts` rejects them defensively if they still appear
- `character-manager.ts` resolves element-anchored start points to the element approach point when possible

Anchored dialogue initiation rules:
- **Unanchored characters**: can initiate dialogue (subject to same-location rule)
- **Main_area anchored** (`anchor.type === "element"`): cannot initiate dialogue at all
- **Region anchored** (`anchor.type === "region"`): can initiate dialogue only within their own region (`currentLocation === anchor.targetId`)

Client-side behavior:
- anchored element characters wander in a smaller radius
- pinned/anchored presentation stays near the movement anchor

Do not weaken these rules unless the user explicitly changes product semantics.

## Dialogue model

Current intent:
- server owns session selection, continuation, and completion
- client owns pixel-level landing and playback presentation
- `talk_to` requires initiator and target to be in the **same location** (`isLegalDirectTalkTarget`)
- `main_area` dialogue: initiator walks to responder's position, maintains ~0.8 body height spacing, facing each other
- when a `main_area` landing succeeds, logical `mainAreaPointId` syncs to the responder point
- dialogue playback is lane-based on the client, so different conversations can play in parallel
- tick progression waits for client playback completion before advancing again (both live and replay)
- bottom dialogue list appears only after the initiator finishes walking and the first bubble is displayed

Relevant files:
- `server/src/simulation/simulation-engine.ts`
- `server/src/simulation/decision-maker.ts`
- `client/src/scenes/WorldScene.ts`
- `client/src/systems/PlaybackController.ts`
- `client/src/systems/CharacterMovement.ts`

## Perception and prompt injection

- **`appearanceHint`**: each character has an optional appearance description visible to others; injected into `formatPerception()` as part of the "能看到的人" list in `prompt-builder.ts`
- **`worldSocialContext`**: a 1-2 sentence world background injected as weak context into reactive decision prompts, dialogue prompts, and dialogue turn prompts via `PromptBuilder`; sourced from `WorldManager.getWorldSocialContext()`

Key files:
- `server/src/simulation/perceiver.ts` — builds `Perception` with `appearanceHint` per visible character
- `server/src/llm/prompt-builder.ts` — formats perception and injects `worldSocialContext`

## God system and sandbox chat

**God system** (`server/src/api/routes/god.ts`, `client/src/ui/panels/GodPanel.tsx`):
- **Broadcast**: sends a message to all characters as an environmental event; optionally writes observation memories
- **Whisper/Dream**: injects a memory into a specific character; whisper content uses second person ("你") as an external voice
- **Profile editing**: character profiles (traits, backstory, etc.) can be modified at runtime via `PATCH /api/characters/:id/profile`

**Sandbox chat** (`server/src/api/routes/sandbox-chat.ts`, `client/src/ui/panels/SandboxChatPanel.tsx`):
- Isolated conversation with a character outside the simulation timeline
- Character is "pulled out" with current memories into a sandbox session
- Conversation does not enter the character's simulation memories

## Time and transitions

Current principles:
- generated worlds normalize to 15-minute ticks
- open scenes derive runtime window from `startTime` + `endTime`
- day/tick persist through `world_global_state` in the active timeline's `state.db`
- `PlaybackController` has configurable `cycleTicks` (default 48); dev mode allows switching tick granularity (15min/30min/1h)
- day transitions use a multi-phase overlay: `ending` (covers screen) → `starting` (shows new day info) → `fade-out` (reveals map)
- `PlaybackController` detects last tick of cycle, emits `scene_ending`, waits for `scene_covered` before requesting next tick from server
- `endOfDayText` and `newDayText` from world design are displayed during transitions

Check these files when changing time semantics:

- `server/src/api/routes/world.ts`

- `client/src/ui/panels/SceneTransition.tsx` — overlay rendering
- `client/src/systems/PlaybackController.ts` — `cycleTicks`, `scene_ending` / `scene_covered`

## Decision model

Current simulation is reactive.

Keep these principles:
- do not reintroduce old daily-plan architecture unless explicitly requested
- lightweight memory retrieval is still the current default
- moving around is a first-class action option, but anchoring and multi-tick action locks remain intact
- if the task is about characters never moving, inspect both `reactive-decision.md` and `perceiver.ts` before changing executors
- energy, hunger, and social needs have been removed; character behavior should evolve freely based on world context

## Verification

After simulation-behavior changes:

1. Test a world with authored regions.
2. Test a world that mostly uses `main_area`.
3. Test anchored characters near interactive elements.
4. Verify dialogue still starts and characters visually approach correctly.
5. Verify idle wandering stays local and believable.
6. Verify 1K and 2K worlds feel consistent in relative movement speed.
7. Verify tick progression works in Run mode (Play / Pause).
8. Verify Replay mode correctly plays back recorded events without server calls.
9. Verify switching between Run and Replay modes does not leave stale state.
10. Verify day transitions work: overlay covers before character repositioning, fades out after.

## Common mistakes

- Mixing logical movement with client-only micro-wandering
- Reintroducing exact-coordinate dialogue logic on the server
- Letting anchored characters receive move actions
- Using fixed-pixel movement/distance values that make 2K/4K worlds feel larger than 1K worlds
- Forgetting that reactive-decision prompt wording can change behavior as much as executor code
- Making server-mutating API calls during replay mode (check `isReplaying` guard)
- Confusing `autoPlay` (live simulation) with `replayAutoPlay` (replay playback) — they are separate state in `PlaybackController`
- Allowing main_area anchored characters to initiate dialogue (they should not)
- Breaking index alignment between `characters.json` and `worldDesign.characters` in the generation pipeline
